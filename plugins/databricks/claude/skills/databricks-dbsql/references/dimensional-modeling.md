# Dimensional Modeling Patterns

Implementation patterns for SCD Type 1/2/6, surrogate keys, batch MERGE upserts, and environment promotion on Delta tables. For architecture decisions (star schema vs OBT, Kimball methodology, clustering key selection), see [best-practices.md](best-practices.md).

For the **streaming** CDC path (event-by-event updates from a pipeline), use Auto CDC in the `databricks-pipelines` skill instead of the MERGE patterns below.

---

## SCD Type Selection

| Type | Strategy | When to use |
|------|----------|-------------|
| **Type 1** | Overwrite in place | History not needed; corrections and label refreshes |
| **Type 2** | Versioned rows (`eff_start_date`, `eff_end_date`, `is_current`) | Audit, regulatory, or revenue re-statement requires history |
| **Type 6** | Type 2 rows + `current_*` column on every row | History required AND frequent current-value lookup without `WHERE is_current = TRUE` overhead |

## Surrogate Keys

### IDENTITY column (preferred for DBSQL batch loads)

Database-assigned; no application-side key generation required.

```sql
CREATE TABLE gold.sales.dim_customer (
  customer_key  BIGINT GENERATED ALWAYS AS IDENTITY,
  customer_id   STRING NOT NULL  COMMENT 'Natural key from source system',
  full_name     STRING,
  city          STRING,
  segment       STRING,
  CONSTRAINT pk_customer PRIMARY KEY (customer_key)
)
CLUSTER BY (customer_key, segment);
```

**Known limitations** — consider the hash-key alternative (below) when any apply:

- **Concurrent transactions are disabled** on a table that has an identity column.
- **Cannot `PARTITION BY`** an identity column.
- **Cannot define an identity column in a CTAS.** Create the table with the identity definition first, then `INSERT` (use `GENERATED BY DEFAULT AS IDENTITY` so existing surrogate values can be back-filled if needed).
- **Cannot `UPDATE`** an identity column value; delete and reinsert instead.
- **Not supported on Auto CDC targets.** Identity columns are not supported on tables that are the target of `APPLY CHANGES` (Auto CDC) in Lakeflow pipelines — use a hash key or omit a pipeline-managed surrogate on those tables.
- **Use only with streaming tables in pipelines.** Identity columns may be recomputed during materialized view refreshes; Databricks recommends using identity columns in pipelines only with streaming tables.

`GENERATED ALWAYS AS IDENTITY` rejects explicit inserts. Use `GENERATED BY DEFAULT AS IDENTITY` when you need to supply values for back-fills or clones (see [Environment Promotion](#environment-promotion-deep-clone)).

### Hash key (deterministic, for distributed loads and pipeline targets)

Use when the surrogate must be stable across pipeline runs without a live sequence, or when any identity-column limitation above applies.

```sql
SELECT
  CAST(xxhash64(customer_id, source_system) AS BIGINT) AS customer_key,
  customer_id,
  full_name,
  city
FROM silver.crm.customers;
```

`xxhash64()` returns a stable 64-bit integer — same input, same key. Validate natural-key uniqueness upstream; 64-bit collisions are negligible but non-zero in very large tables.

## SCD Type 1 — Overwrite

Single MERGE; no history rows.

```sql
MERGE INTO gold.sales.dim_product AS t
USING silver.catalog.products AS s
ON t.product_id = s.product_id
WHEN MATCHED AND (
  t.product_name != s.product_name OR t.list_price != s.list_price
) THEN UPDATE SET
  t.product_name = s.product_name,
  t.list_price   = s.list_price,
  t.updated_at   = current_timestamp()
WHEN NOT MATCHED THEN INSERT (product_id, product_name, list_price, updated_at)
  VALUES (s.product_id, s.product_name, s.list_price, current_timestamp());
```

## SCD Type 2 — History Tracking

### Dimension DDL

```sql
CREATE TABLE gold.sales.dim_customer (
  customer_key   BIGINT GENERATED ALWAYS AS IDENTITY,
  customer_id    STRING NOT NULL  COMMENT 'Natural key',
  full_name      STRING,
  city           STRING,
  segment        STRING,
  eff_start_date TIMESTAMP NOT NULL,
  eff_end_date   TIMESTAMP,           -- NULL = current version
  is_current     BOOLEAN NOT NULL,
  CONSTRAINT pk_customer PRIMARY KEY (customer_key)
)
CLUSTER BY (customer_key, customer_id)
COMMENT 'Customer dimension with SCD Type 2 history';
```

### Two-pass MERGE

**Scope:** this pattern handles changed rows (new version) and net-new rows (first insert). It does **not** handle deletes — rows that disappear from the source are not expired (`is_current` is not flipped to `FALSE`). Add explicit delete handling if the source can drop rows.

A single MERGE cannot expire an existing row and insert a new version with the same natural key in one pass. Use two sequential statements.

```sql
-- Pass 1: expire changed current rows
MERGE INTO gold.sales.dim_customer AS t
USING (
  SELECT s.*
  FROM silver.crm.customer_changes s
  JOIN gold.sales.dim_customer d
    ON s.customer_id = d.customer_id AND d.is_current = TRUE
  WHERE s.full_name != d.full_name OR s.city != d.city
) AS changed
ON t.customer_id = changed.customer_id AND t.is_current = TRUE
WHEN MATCHED THEN UPDATE SET
  t.eff_end_date = current_timestamp(),
  t.is_current   = FALSE;

-- Pass 2: insert new versions (changed rows, now expired) + net-new rows
INSERT INTO gold.sales.dim_customer
  (customer_id, full_name, city, segment, eff_start_date, eff_end_date, is_current)
SELECT
  s.customer_id, s.full_name, s.city, s.segment,
  current_timestamp(), NULL, TRUE
FROM silver.crm.customer_changes s
LEFT JOIN gold.sales.dim_customer t
  ON s.customer_id = t.customer_id AND t.is_current = TRUE
WHERE t.customer_id IS NULL;  -- NULL after expiry (changed) OR never existed (net-new)
```

Run Pass 1 before Pass 2. Out-of-order leaves two rows with `is_current = TRUE` for the same natural key.

For querying the resulting history table (`__END_AT IS NULL`, point-in-time joins, etc.), see `scd-2-querying.md` in the `databricks-pipelines` skill.

## SCD Type 6 — Hybrid (Type 2 rows + current-value columns)

SCD Type 6 stores versioned rows like Type 2 and also keeps the *current* attribute value on every row — including historical ones — so callers can read the current value without filtering on `is_current`.

### Extra DDL columns

```sql
current_full_name STRING COMMENT 'Latest value — refreshed on all rows for this customer',
current_city      STRING,
```

### Three-pass update

Pass 1 and Pass 2 are identical to SCD Type 2. Pass 3 refreshes `current_*` columns on all history rows.

```sql
-- Pass 1: expire changed current rows  (same as Type 2)
-- Pass 2: insert new versions           (same as Type 2)

-- Pass 3: propagate current values to all history rows for changed entities
UPDATE gold.sales.dim_customer AS t
SET t.current_full_name = s.full_name,
    t.current_city      = s.city
FROM silver.crm.customer_changes s
WHERE t.customer_id = s.customer_id;
```

Pass 3 touches all history rows for each changed entity. If history rows per entity are large, a `dim_customer_current` materialized view (`WHERE is_current = TRUE`) is a lighter alternative that avoids the column refresh entirely.

## Table Layout

### Clustering keys

| Layer | Dimension tables | Fact tables |
|-------|-----------------|-------------|
| Gold | `CLUSTER BY (surrogate_key, common_filter_col)` | `CLUSTER BY (date_key, high-traffic FK)` |

Limit to 2–4 keys; 2 keys often outperform 4 for tables under 10 TB. See [best-practices.md](best-practices.md) for full selection guidance.

### Predictive Optimization

Automates OPTIMIZE and VACUUM for Unity Catalog **managed** tables (Delta + Iceberg) in non-OpenSharing catalogs. It is **enabled by default** for accounts created on or after 2024-11-11; a gradual rollout to older accounts is in progress (expected complete ~Aug 2026). For older accounts, or to override at a finer scope, use the explicit opt-in:

```sql
-- Enable / disable / inherit at catalog or schema level
ALTER CATALOG gold { ENABLE | DISABLE | INHERIT } PREDICTIVE OPTIMIZATION;
ALTER SCHEMA gold.sales { ENABLE | DISABLE | INHERIT } PREDICTIVE OPTIMIZATION;
-- or at table level:
ALTER TABLE gold.sales.dim_customer ENABLE PREDICTIVE OPTIMIZATION;
```

`INHERIT` resets a table or schema to follow its parent's setting. PO does not apply to external tables or tables in OpenSharing (read-only recipient) catalogs.

Collect statistics on surrogate and foreign keys after initial load or major refreshes:

```sql
ANALYZE TABLE gold.sales.dim_customer COMPUTE STATISTICS FOR COLUMNS customer_key, segment;
ANALYZE TABLE gold.sales.fact_orders  COMPUTE STATISTICS FOR COLUMNS date_key, customer_key, product_key;
```

## Environment Promotion (Deep Clone)

`DEEP CLONE` copies data files, schema, and `TBLPROPERTIES` to a new independent table. Re-running against the same target copies only new/changed files (incremental). The clone does **not** inherit source table history — time travel on the clone starts fresh from the point of cloning.

```sql
-- Promote prod dimension to dev (creates table if absent; incremental on re-run)
CREATE OR REPLACE TABLE dev_catalog.sales.dim_customer
DEEP CLONE prod_catalog.sales.dim_customer;
```

Writes to `dev_catalog.sales.dim_customer` do not affect the source.

**Managed Iceberg tables support `DEEP CLONE`; `SHALLOW CLONE` is not supported on Iceberg.** Additional requirements for Iceberg targets: Predictive Optimization must be enabled on the target catalog or schema, and DBR 16.4 LTS+ is required. Table properties set on the source are copied, but Unity Catalog tags and descriptions are not — apply those with `ALTER TABLE` afterward.

```sql
-- Iceberg deep clone (requires PO on target catalog/schema, DBR 16.4+)
CREATE OR REPLACE TABLE dev_catalog.sales.dim_customer_iceberg
DEEP CLONE prod_catalog.sales.dim_customer_iceberg;
```

## Common Issues

| Issue | Cause / Fix |
|-------|-------------|
| Two rows with `is_current = TRUE` for same natural key | Pass 1 (expire) and Pass 2 (insert) ran out of order, or expired row was not filtered from Pass 2 `LEFT JOIN`. Ensure `AND t.is_current = TRUE` in the Pass 2 join predicate. |
| `GENERATED ALWAYS AS IDENTITY` rejects insert | Cannot supply explicit values. Switch to `GENERATED BY DEFAULT AS IDENTITY` for CTAS or DEEP CLONE scenarios. |
| Hash collision on `xxhash64` | Two natural keys produce the same 64-bit hash. Validate natural key uniqueness; consider appending a disambiguator column to the hash input. |
| Pass 3 (Type 6 `current_*` refresh) is slow | Many history rows per entity. Replace with a `CREATE OR REFRESH MATERIALIZED VIEW dim_customer_current AS SELECT * FROM dim_customer WHERE is_current = TRUE`. |
| `CLUSTER BY` fails with `DELTA_CLUSTERING_COLUMNS_DATATYPE_NOT_SUPPORTED` | A clustering key is `BOOLEAN`, `ARRAY`, `MAP`, `STRUCT`, or `BINARY`. Use numeric / string / date / timestamp columns only. |
| Iceberg `DEEP CLONE` fails | Target catalog/schema does not have Predictive Optimization enabled, or DBR < 16.4 LTS. Enable PO on the target catalog/schema and use DBR 16.4+. |
| Time-travel queries fail after DEEP CLONE | Deep clones do not copy table history — the clone has an independent snapshot history. Queries referencing source-table snapshots older than the clone point will fail on the clone. |
